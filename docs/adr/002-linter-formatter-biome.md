# ADR-002: リンター/フォーマッター: Biome

## ステータス

Accepted

## コンテキスト

コードの品質維持と一貫したフォーマットのために、リンターとフォーマッターを導入する必要がある。
従来は ESLint + Prettier の組み合わせが一般的だったが、新しいツールも登場している。

## 決定

Biome を採用する。

## 選択肢

### 選択肢 1: Biome

- メリット
  - リンターとフォーマッターが統合されており、設定がシンプル
  - Rust 製で非常に高速（ESLint + Prettier の 10〜100 倍）
  - ESLint / Prettier との互換性が高く、移行が容易
  - 単一の設定ファイル（biome.json）で管理可能
  - プラグイン不要で TypeScript、JSX をサポート
- デメリット
  - ESLint と比較してルールの数が少ない
  - プラグインエコシステムが未発達
  - 比較的新しいツールのため、情報が少ない

### 選択肢 2: ESLint + Prettier

- メリット
  - 長年の実績があり、情報が豊富
  - 豊富なプラグインエコシステム
  - 細かいカスタマイズが可能
- デメリット
  - 2 つのツールの設定が必要で複雑
  - 実行速度が遅い
  - ESLint と Prettier の競合を解決する必要がある

### 選択肢 3: ESLint (Flat Config) + ESLint Stylistic

- メリット
  - ESLint のみで完結
  - Flat Config による新しい設定方式
- デメリット
  - 設定が複雑
  - Prettier ほどのフォーマット品質ではない

## 理由

Biome を採用する理由は以下の通り：

1. **シンプルさ**: 1 つのツールでリントとフォーマットを完結できる
2. **高速性**: Rust 製で非常に高速、大規模プロジェクトでもストレスなく実行可能
3. **設定の簡素化**: 単一の設定ファイルで管理でき、プラグインの互換性問題がない
4. **十分なルール**: 一般的なプロジェクトで必要なルールは網羅されている

新規プロジェクトであり、ESLint のプラグインエコシステムに依存する必要がないため、Biome の採用が適切と判断した。

## 結果

- biome.json で統一的なルールを設定
- `pnpm check` コマンドでリントとフォーマットを同時実行
- エディタ拡張機能を導入し、保存時に自動フォーマット
